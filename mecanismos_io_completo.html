Capítulo 01 / 03

Buffering

Definición formal

El buffering es una técnica en la que el sistema operativo reserva una región de memoria principal (RAM) para almacenar datos transitoriamente mientras se transfieren entre dos entidades que operan a velocidades distintas: un productor que genera datos y un consumidor que los procesa. El buffer actúa como zona de amortiguación, permitiendo que ambas partes avancen a su propio ritmo sin bloquearse mutuamente.

¿Por qué existe el buffering?

Imagina un proceso que lee un archivo de disco y lo procesa. Sin buffer, cada vez que el proceso necesita un byte, lanza una llamada al sistema que bloquea la CPU hasta que el disco entrega ese byte. Con un disco duro clásico que tarda 10 ms por acceso, y un proceso que necesita millones de bytes, el tiempo de espera sería astronómico.

Con buffering, el sistema operativo llena un bloque completo (por ejemplo, 4 KB o 64 KB) de una sola vez. Luego el proceso consume ese bloque desde la RAM a velocidad de nanosegundos mientras el sistema prepara el siguiente bloque en paralelo. La CPU nunca espera al disco — espera a la RAM, que es miles de veces más rápida.

Analogía del mundo real

Imagina que traes agua de un pozo con un balde. Sin buffer: cada vez que tienes sed, caminas al pozo, traes un sorbo, bebes, vuelves. Con buffer: traes un balde lleno (el buffer), bebes de él cómodamente, y mientras lo usas, alguien más puede ir llenando otro balde. Nunca esperas — la fuente lenta (el pozo) y el consumidor rápido (tu sed) operan desacoplados.

Tipos de buffering y cómo funcionan

Single Buffer

Un único bloque en RAM. El productor llena el buffer; el consumidor lo vacía. Mientras el consumidor trabaja, el productor espera. Simple pero ineficiente: hay un período de "pausa" entre llenados.

Double Buffer

Dos bloques alternados (ping-pong). Mientras el consumidor procesa el bloque A, el productor llena el bloque B. Cuando A termina, se intercambian roles. Elimina casi por completo el tiempo de espera.

Circular Buffer (Ring Buffer)

N bloques organizados en anillo con dos punteros: escritura y lectura. Solución estándar en streaming de audio, video, redes y drivers de dispositivo. Máxima eficiencia para flujos continuos.

Buffer de E/S de Kernel

El kernel mantiene sus propios buffers entre el espacio de usuario y los dispositivos. Cuando una aplicación llama a write(), los datos van primero al buffer del kernel y luego al dispositivo de forma asíncrona.

La semántica de copia: un detalle crítico

Existe un concepto que los libros básicos omiten pero que es fundamental para la correctitud del sistema: la copy semantics (semántica de copia).

Cuando un proceso llama a write(buffer_usuario, 1024), el sistema operativo no envía inmediatamente ese buffer al dispositivo. En cambio, copia los datos al buffer interno del kernel. ¿Por qué esto importa?

⚠
Escenario sin semántica de copia

Un proceso llama a write() con su propio buffer. Antes de que el kernel termine de escribir al disco, el proceso modifica ese mismo buffer en memoria. Sin copia, el dato escrito al disco sería la versión modificada, no la original. Con la copia al kernel buffer, la versión correcta está garantizada.

Buffering en la red: el caso ECN / DCTCP

En centros de datos modernos el buffering va mucho más allá del disco. Los switches de red tienen buffers internos finitos. Cuando llegan paquetes más rápido de lo que salen (congestión), el buffer del switch se llena. El comportamiento clásico era: cuando se llena → descarta paquetes → el emisor detecta la pérdida → reduce velocidad. Este ciclo causa incast collapse: colapso masivo del throughput.

DCTCP (Data Center TCP) usa Explicit Congestion Notification (ECN): cuando el buffer del switch supera un umbral (antes de llenarse), marca los paquetes con el bit CE (Congestion Experienced). El receptor notifica al emisor, que reduce gradualmente su velocidad. Resultado: los buffers nunca llegan a llenarse completamente, la latencia se mantiene baja, y se evita el colapso.

Capítulo 02 / 03

Spooling

Definición formal

El spooling (Simultaneous Peripheral Operation On-Line) es una especialización del buffering que utiliza el disco secundario como buffer de gran capacidad. Su propósito principal no es sólo compensar la diferencia de velocidades, sino lograr el desacoplamiento completo entre el proceso que genera los datos y el dispositivo que los consume — permitiendo que múltiples trabajos coexistan en cola mientras el dispositivo lento procesa cada uno en orden.

La diferencia clave con el buffering

Esta distinción es tan importante que merece una comparación directa:

Criterio Buffering Spooling
¿Qué desacopla? E/S y cómputo del mismo proceso Generación y consumo entre procesos distintos
¿Dónde almacena? RAM (volátil, nanosegundos) Disco (persistente, milisegundos)
¿Qué guarda? Fragmentos de un flujo de datos Trabajos completos (documentos, lotes)
Fallo de energía Pérdida total de datos Recuperación completa del trabajo
Multiprogramación Dentro de un proceso Entre múltiples procesos simultáneos

¿Por qué las impresoras necesitan spooling obligatoriamente?

La impresora encarna perfectamente el problema que resuelve el spooling. Tiene cuatro propiedades que la hacen un caso especial:

① Dispositivo exclusivo

Una impresora no puede intercalar bytes de dos documentos distintos — el resultado serían páginas mezcladas ilegibles. Solo puede atender un trabajo completo a la vez. El spooling garantiza atomicidad: cada trabajo se entrega entero.

② Velocidad abismal

Una impresora láser imprime 30-50 páginas/minuto ≈ 1-2 MB/s. La CPU genera datos a GB/s. Sin spooling, el proceso quedaría bloqueado durante minutos esperando que la impresora digiera el documento.

③ Múltiples usuarios

En entornos multiusuario (oficinas, universidades), varias personas quieren imprimir simultáneamente. El spooler serializa los trabajos en una cola FIFO: todos "terminan de imprimir" inmediatamente desde su perspectiva.

④ Persistencia necesaria

Si el sistema se reinicia a mitad de una impresión, el trabajo debe sobrevivir. Como el spool vive en disco, el spooler puede retomar exactamente donde dejó al reiniciarse.

PrintNightmare: cuando el spooler se convierte en vector de ataque

El spooler de Windows opera con privilegios de SYSTEM (el nivel más alto del sistema operativo) porque necesita acceder a hardware, instalar drivers en modo kernel y gestionar recursos privilegiados. Esta necesidad arquitectónica crea una superficie de ataque peligrosa.

🔴
CVE-2021-34527 — PrintNightmare

La vulnerabilidad surgió de un error en la función RpcAddPrinterDriverEx(): cualquier usuario autenticado del dominio podía invocarla para que el spooler (con privilegios SYSTEM) cargara un DLL malicioso desde una ruta de red controlada por el atacante. Resultado: ejecución de código arbitrario con los máximos privilegios del sistema.

Junio 2021

Investigadores de Sangfor descubren el zero-day y publican accidentalmente un PoC funcional en GitHub.

1 Julio 2021

Microsoft asigna CVE-2021-34527. El PoC ya se ha copiado en 34+ repositorios. Afecta a todas las versiones de Windows desde Win 7 hasta Win Server 2022.

6 Julio 2021

Microsoft publica parche de emergencia (out-of-band). Investigadores descubren que el parche solo cubre el componente RCE, no la escalada de privilegios local.

Post-2021

Microsoft introduce RestrictDriverInstallationToAdministrators, deshabilita la impresión remota entrante por defecto, y aísla procesos de renderizado. Desde 2021: cero nuevas vulnerabilidades RCE en el spooler — solo escaladas de privilegios menores.

La lección arquitectónica es profunda: cualquier servicio que combine alta privilegiación + superficie de ataque amplia + compatibilidad con código de terceros (drivers de fabricantes de impresoras) es estructuralmente riesgoso.

Caching

Definición formal

El caching es la práctica de mantener copias de datos frecuentemente accedidos en un nivel de almacenamiento más rápido, aprovechando los principios de localidad temporal y espacial. A diferencia del buffer (que puede ser la única copia del dato), el caché siempre duplica un dato que ya existe en otro nivel más lento. Opera de manera completamente transparente al proceso: ninguna aplicación necesita saber si está usando el caché o el almacenamiento original.

Los dos principios fundamentales del caching

Localidad Temporal

Si un dato fue accedido recientemente, probablemente volverá a accederse pronto. Ejemplo: la variable de control de un bucle for (i=0; i<n; i++) se accede millones de veces consecutivas. El caché la mantiene "caliente".

Localidad Espacial

Si un dato fue accedido, los datos cercanos en memoria probablemente también se accederán. Ejemplo: al leer el elemento array[5], el caché carga también array[6], array[7]... porque es probable que el código también los necesite.

El kernel de Linux usa la RAM como caché del disco (page cache).

El Page Cache de Linux: cómo el SO usa la RAM como caché de disco

En Linux moderno (kernel ≥ 2.4), el page cache es el mecanismo central de caching de E/S. Cuando lees un archivo por primera vez, el kernel lo carga del disco y lo guarda en RAM. La segunda vez que lees ese archivo — incluso desde otro proceso — el kernel sirve los datos directamente desde RAM sin tocar el disco.

Algoritmos de reemplazo: ¿qué bloque se expulsa cuando el caché está lleno?

LRU — Least Recently Used

Se expulsa el bloque que lleva más tiempo sin accederse. Aprovecha la localidad temporal. Es el algoritmo más común en sistemas de archivos Linux (ext4, XFS, Btrfs). Requiere mantener un registro de tiempo de último acceso.

LFU — Least Frequently Used

Se expulsa el bloque menos frecuentemente accedido. Útil cuando ciertos datos se acceden repetidamente durante períodos largos. Problema: datos "envejecidos" con muchos accesos históricos pueden ocupar espacio innecesariamente.

FIFO — First In, First Out

El primer bloque en entrar es el primero en salir. Simple de implementar pero ignora completamente el patrón de acceso. Puede expulsar bloques muy activos si llevan mucho tiempo en caché.

ARC — Adaptive Replacement Cache

Algoritmo adaptativo que combina LRU y LFU dinámicamente. Usado en ZFS. Se adapta automáticamente a la carga de trabajo actual, comportándose como LRU para accesos únicos y LFU para accesos repetidos.

El cuello de botella migró al software: el problema NVMe

Una investigación reciente de USENIX FAST 2025 reveló algo contraintuitivo: con SSDs NVMe ultra-rápidos, el cuello de botella ya no es el hardware sino el propio page cache del kernel. La asignación de páginas de memoria para el caché consume entre un 24% y 36% de todos los ciclos de CPU durante operaciones de E/S pesada. El software del kernel, diseñado décadas atrás para discos lentos, se convirtió en el nuevo limitante.

💡
Consecuencia práctica

Bases de datos como PostgreSQL y MySQL permiten usar O_DIRECT para saltarse el page cache completamente y hacer E/S directa. Esto evita la doble copia (disco→page cache→buffer propio de la BD) y la sobrecarga del kernel, a costa de perder la transparencia del caching del SO.

Buffering Ejercicio 1 — Cálculo de tiempo con y sin double buffering
Enunciado Un proceso lee un archivo de 10 MB del disco y procesa cada bloque de 1 MB. El disco tarda 50 ms en leer cada bloque (tiempo de llenado del buffer). El procesador tarda 30 ms en procesar cada bloque (tiempo de vaciado).

a) ¿Cuánto tiempo tarda el proceso completo con single buffering?
b) ¿Cuánto tiempo tarda con double buffering?
c) ¿Cuál es el ahorro porcentual?
Solución paso a paso

Datos del problema:

• Tamaño del archivo: 10 MB → 10 bloques de 1 MB
• Tiempo de lectura por bloque (disco → buffer): TE/S = 50 ms
• Tiempo de proceso por bloque (CPU consume buffer): TCPU = 30 ms
• El disco es más lento: TE/S > TCPU

a) Single Buffering:

Con un solo buffer, el proceso es estrictamente secuencial: leer bloque → procesar bloque → leer siguiente. No hay solapamiento posible.

T_single = N × (T_E/S + T_CPU) = 10 × (50 ms + 30 ms) = 10 × 80 ms = 800 ms
Bloque 1: [─ DISCO 50ms ─][─ CPU 30ms ─] Bloque 2: [─ DISCO 50ms ─][─ CPU 30ms ─] Bloque 3: [─ DISCO 50ms ─][─ CPU 30ms ─] ├──────────────┤ 80 ms × 10 = 800 ms total

b) Double Buffering:

Con dos buffers, el disco llena el buffer B mientras la CPU procesa el buffer A. El tiempo de cada ciclo es el máximo de los dos (el proceso más lento manda el ritmo).

T_double = T_E/S_primer_bloque + (N-1) × max(T_E/S, T_CPU) + T_CPU_último
= 50 ms + 9 × max(50, 30) ms + 30 ms
= 50 ms + 9 × 50 ms + 30 ms
= 50 + 450 + 30 = 530 ms
DISCO: [leer A──50ms][leer B──50ms][leer A──50ms][leer B──50ms]... CPU: [proc A─30ms─][proc B─30ms─][proc A─30ms─]... │←── solapamiento ──→│ El cuello de botella es el disco (50ms > 30ms), cada ciclo dura 50ms. 10 bloques: 50ms (inicial) + 9 × 50ms + 30ms (último CPU) = 530 ms

c) Ahorro porcentual:

Ahorro = (800 - 530) / 800 × 100 = 270/800 × 100 = 33.75%

El double buffering reduce el tiempo en un 33.75%. El ahorro es mayor cuando TCPU ≈ TE/S; si ambos fueran iguales (50 ms), el ahorro llegaría al 50%.

Spooling Ejercicio 2 — Análisis de cola de impresión y tiempo de bloqueo
Enunciado En una oficina, 3 usuarios envían trabajos de impresión simultáneamente:
• Usuario A: documento de 5 páginas (generación: 2 s, impresión: 15 s)
• Usuario B: documento de 10 páginas (generación: 4 s, impresión: 30 s)
• Usuario C: documento de 2 páginas (generación: 1 s, impresión: 6 s)

a) Sin spooling: ¿cuánto tiempo cada usuario espera bloqueado?
b) Con spooling (cola FIFO orden A→B→C): ¿cuánto espera cada usuario bloqueado?
c) ¿Cuál es el tiempo total hasta que cada usuario "termina" desde su perspectiva?
Solución paso a paso

a) Sin spooling — acceso exclusivo secuencial:

Sin spooling, la impresora es un recurso exclusivo. El primer usuario que la toma monopoliza el dispositivo completo. Los demás esperan bloqueados en su proceso hasta que puedan acceder.

t=0s: A comienza a imprimir directamente (monopoliza impresora) t=0→15s: A imprime. B y C generaron su doc pero ESPERAN BLOQUEADOS. t=15s: A termina. B ocupa la impresora. t=15→45s: B imprime. C ESPERA BLOQUEADO. t=45s: B termina. C imprime. t=45→51s: C imprime. t=51s: Sistema termina.

• Usuario A: bloqueado 15 s (duración de su propia impresión)
• Usuario B: bloqueado 15 + 30 = 45 s (espera A + su propia impresión)
• Usuario C: bloqueado 15 + 30 + 6 = 51 s

b) Con spooling — los usuarios NO se bloquean esperando la impresora:

Con spooling, cada usuario escribe su trabajo a disco (muy rápido) y queda libre inmediatamente. El spooler gestiona la cola en background.

t=0s: A envía trabajo al spooler → tarda 2s en generar+escribir al spool B envía trabajo al spooler → tarda 4s C envía trabajo al spooler → tarda 1s Desde perspectiva de cada usuario: Usuario A: bloqueado solo 2s (tiempo de generar + escribir a spool) Usuario B: bloqueado solo 4s Usuario C: bloqueado solo 1s El spooler trabaja en background (cola FIFO A→B→C): t=2s: Spooler inicia impresión de A t=17s: Spooler inicia impresión de B t=47s: Spooler inicia impresión de C t=53s: Cola de impresión vacía

c) Tiempo hasta "terminar" desde la perspectiva de cada usuario:

Usuario Sin Spooling Con Spooling Ahorro
A 15 s bloqueado 2 s bloqueado 86.7% menos
B 45 s bloqueado 4 s bloqueado 91.1% menos
C 51 s bloqueado 1 s bloqueado 98.0% menos

El spooling libera la CPU de los procesos usuario casi de inmediato. El trabajo de impresión real ocurre en background, gestionado por el daemon del spooler de forma completamente independiente.

Comparativa Ejercicio 5 — Identificación del mecanismo correcto por escenario
Enunciado Para cada uno de los siguientes escenarios, identifica qué mecanismo(s) se están usando (buffering, spooling, caching, o combinación) y justifica tu respuesta:

a) Netflix precarga los próximos 30 segundos de video mientras ves la escena actual.
b) Un servidor web guarda en memoria la página de inicio que recibe 10,000 visitas/minuto.
c) Cuando imprimes desde Word, el icono de impresora aparece en la barra de tareas y Word queda libre inmediatamente.
d) El compilador GCC lee el archivo fuente .c y lo procesa bloque a bloque sin cargar todo en memoria.
e) Tu navegador web guarda imágenes de sitios visitados en una carpeta local para no descargarlas la próxima vez.
Solución — Análisis de cada escenario

a) Netflix precarga los próximos 30 segundos:

→ Buffering (específicamente: buffer circular / ring buffer)

Se está amortiguando la diferencia de velocidad entre la red (variable, intermitente) y la reproducción (constante, tiempo real). Los segmentos futuros se almacenan en un buffer en RAM. El reproductor consume del buffer mientras la red llena el buffer con segmentos posteriores. Si la red cae momentáneamente, el buffer mantiene la reproducción sin interrupciones.

b) El servidor web guarda la página de inicio en memoria:

→ Caching

La página de inicio es un dato que existe en disco (fuente original) y se duplica en RAM para servir accesos repetidos más rápidamente. Cumple exactamente la definición: copia de un dato existente en almacenamiento más rápido. Nota clave: si el servidor modificara el HTML, debe invalidar la entrada del caché (cache invalidation).

c) Imprimir desde Word y quedar libre inmediatamente:

→ Spooling

Word entrega el trabajo al spooler del sistema operativo (spoolsv.exe en Windows, CUPS en Linux). El spooler escribe el trabajo en disco (archivo .SPL) y retorna el control a Word casi inmediatamente. Word no espera a que la impresora física termine. La cola de impresión visible en la barra de tareas es la interfaz del spooler mostrando el estado de la cola.

d) GCC lee el archivo .c bloque a bloque:

→ Buffering + Caching (combinación)

El compilador usa buffering de E/S: en lugar de leer el archivo carácter a carácter (llamada al sistema por cada byte), el kernel lee bloques enteros al buffer del kernel. Adicionalmente, si el archivo fue abierto recientemente, el page cache del SO puede servirlo directamente desde RAM sin tocar el disco.

e) El navegador guarda imágenes localmente:

→ Caching (HTTP cache / browser cache)

Las imágenes en disco local son copias de datos que originalmente viven en servidores remotos. La próxima visita al sitio, el navegador sirve las imágenes desde la copia local en lugar de descargarlas de nuevo. El original sigue existiendo en el servidor. El caché del navegador respeta las cabeceras HTTP (Cache-Control, ETag, Expires) para saber cuándo invalidar una entrada.

Seguridad Ejercicio 6 — Análisis de PrintNightmare desde perspectiva técnica
Enunciado A partir de la vulnerabilidad CVE-2021-34527 (PrintNightmare):

a) Explica en términos técnicos exactamente qué función del spooler fue explotada y por qué.
b) ¿Por qué el spooler necesita correr con privilegios SYSTEM? ¿No podría correr con privilegios de usuario normal?
c) Propón una arquitectura alternativa que resolvería el problema de seguridad sin eliminar el spooling.
Solución

a) La función explotada: RpcAddPrinterDriverEx()

Esta función del spooler permite instalar un driver de impresora en el sistema. Los drivers de impresora son DLLs que se cargan en el proceso del spooler (que corre como SYSTEM) para comunicarse con hardware específico de cada fabricante.

El fallo: la función no validaba correctamente que el usuario que la invocaba tuviera privilegios administrativos. Cualquier usuario de dominio autenticado podía llamar a RpcAddPrinterDriverEx() con una ruta a un DLL malicioso en un servidor SMB controlado por el atacante. El spooler cargaba ese DLL y lo ejecutaba con privilegios SYSTEM.

El vector: la función estaba expuesta vía RPC (Remote Procedure Call) — accesible desde la red, no solo localmente. Un atacante con credenciales básicas de dominio podía explotar servidores sin tener acceso físico.

b) ¿Por qué el spooler necesita SYSTEM?

El spooler realiza operaciones que requieren privilegios de sistema:

  1. Instalar y cargar drivers de impresora en modo kernel (código que corre con máximos privilegios para comunicarse con hardware).

  2. Escribir en directorios del sistema (C:\Windows\System32\spool\) para almacenar los trabajos en cola.

  3. Gestionar colas compartidas accesibles por todos los usuarios del sistema.

  4. Aceptar conexiones de red (RPC) para impresión remota, lo cual requiere privilegios de red elevados.

Si corriera como usuario normal, no podría instalar drivers ni acceder a las rutas del sistema necesarias.

c) Arquitectura alternativa — separación de privilegios:

La solución moderna usa el principio de least privilege (mínimo privilegio) con separación de componentes:

┌─────────────────────────────────────────────────────────────┐ │ PROCESO DE USUARIO (sin privilegios) │ │ • Genera el trabajo de impresión │ │ • Escribe al spool via API segura │ └──────────────────────────────┬──────────────────────────────┘ │ IPC segura (named pipe) ▼ ┌─────────────────────────────────────────────────────────────┐ │ SPOOLER BROKER (privilegios limitados) │ │ • Gestiona cola de trabajos en disco │ │ • NO carga drivers, NO acepta RPC de red │ │ • Valida y sanitiza trabajos │ └──────────────────────────────┬──────────────────────────────┘ │ canal interno auditado ▼ ┌─────────────────────────────────────────────────────────────┐ │ DRIVER HOST (proceso aislado, SYSTEM solo para su tarea) │ │ • Único componente que carga drivers de fabricante │ │ • Sandboxed: no tiene acceso a red ni al sistema de archivos│ │ • Si explota: impacto contenido al proceso aislado │ └─────────────────────────────────────────────────────────────┘

Este diseño es el que Microsoft adoptó parcialmente post-2021 con el modelo "Print Isolation" en Windows 11. El principio general se llama privilege separation o compartmentalization — dividir un servicio privilegiado en subcomponentes con el mínimo privilegio necesario para cada tarea.
