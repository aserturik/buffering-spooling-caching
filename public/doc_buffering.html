Buffering

Definición formal

El buffering es una técnica en la que el sistema operativo reserva una región de memoria principal (RAM) para almacenar datos transitoriamente mientras se transfieren entre dos entidades que operan a velocidades distintas: un productor que genera datos y un consumidor que los procesa. El buffer actúa como zona de amortiguación, permitiendo que ambas partes avancen a su propio ritmo sin bloquearse mutuamente.

¿Por qué existe el buffering?

Imagina un proceso que lee un archivo de disco y lo procesa. Sin buffer, cada vez que el proceso necesita un byte, lanza una llamada al sistema que bloquea la CPU hasta que el disco entrega ese byte. Con un disco duro clásico que tarda 10 ms por acceso, y un proceso que necesita millones de bytes, el tiempo de espera sería astronómico.

Con buffering, el sistema operativo llena un bloque completo (por ejemplo, 4 KB o 64 KB) de una sola vez. Luego el proceso consume ese bloque desde la RAM a velocidad de nanosegundos mientras el sistema prepara el siguiente bloque en paralelo. La CPU nunca espera al disco — espera a la RAM, que es miles de veces más rápida.

Analogía del mundo real

Imagina que traes agua de un pozo con un balde. Sin buffer: cada vez que tienes sed, caminas al pozo, traes un sorbo, bebes, vuelves. Con buffer: traes un balde lleno (el buffer), bebes de él cómodamente, y mientras lo usas, alguien más puede ir llenando otro balde. Nunca esperas — la fuente lenta (el pozo) y el consumidor rápido (tu sed) operan desacoplados.

Tipos de buffering y cómo funcionan

Single Buffer

Un único bloque en RAM. El productor llena el buffer; el consumidor lo vacía. Mientras el consumidor trabaja, el productor espera. Simple pero ineficiente: hay un período de "pausa" entre llenados.

Double Buffer

Dos bloques alternados (ping-pong). Mientras el consumidor procesa el bloque A, el productor llena el bloque B. Cuando A termina, se intercambian roles. Elimina casi por completo el tiempo de espera.

Circular Buffer (Ring Buffer)

N bloques organizados en anillo con dos punteros: escritura y lectura. Solución estándar en streaming de audio, video, redes y drivers de dispositivo. Máxima eficiencia para flujos continuos.

Buffer de E/S de Kernel

El kernel mantiene sus propios buffers entre el espacio de usuario y los dispositivos. Cuando una aplicación llama a write(), los datos van primero al buffer del kernel y luego al dispositivo de forma asíncrona.

La semántica de copia: un detalle crítico

Existe un concepto que los libros básicos omiten pero que es fundamental para la correctitud del sistema: la copy semantics (semántica de copia).

Cuando un proceso llama a write(buffer_usuario, 1024), el sistema operativo no envía inmediatamente ese buffer al dispositivo. En cambio, copia los datos al buffer interno del kernel. ¿Por qué esto importa?

⚠
Escenario sin semántica de copia

Un proceso llama a write() con su propio buffer. Antes de que el kernel termine de escribir al disco, el proceso modifica ese mismo buffer en memoria. Sin copia, el dato escrito al disco sería la versión modificada, no la original. Con la copia al kernel buffer, la versión correcta está garantizada.

Buffering en la red: el caso ECN / DCTCP

En centros de datos modernos el buffering va mucho más allá del disco. Los switches de red tienen buffers internos finitos. Cuando llegan paquetes más rápido de lo que salen (congestión), el buffer del switch se llena. El comportamiento clásico era: cuando se llena → descarta paquetes → el emisor detecta la pérdida → reduce velocidad. Este ciclo causa incast collapse: colapso masivo del throughput.

DCTCP (Data Center TCP) usa Explicit Congestion Notification (ECN): cuando el buffer del switch supera un umbral (antes de llenarse), marca los paquetes con el bit CE (Congestion Experienced). El receptor notifica al emisor, que reduce gradualmente su velocidad. Resultado: los buffers nunca llegan a llenarse completamente, la latencia se mantiene baja, y se evita el colapso.
