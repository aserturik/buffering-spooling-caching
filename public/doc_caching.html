Caching

Definición formal

El caching es la práctica de mantener copias de datos frecuentemente accedidos en un nivel de almacenamiento más rápido, aprovechando los principios de localidad temporal y espacial. A diferencia del buffer (que puede ser la única copia del dato), el caché siempre duplica un dato que ya existe en otro nivel más lento. Opera de manera completamente transparente al proceso: ninguna aplicación necesita saber si está usando el caché o el almacenamiento original.

Los dos principios fundamentales del caching

Localidad Temporal

Si un dato fue accedido recientemente, probablemente volverá a accederse pronto. Ejemplo: la variable de control de un bucle for (i=0; i<n; i++) se accede millones de veces consecutivas. El caché la mantiene "caliente".

Localidad Espacial

Si un dato fue accedido, los datos cercanos en memoria probablemente también se accederán. Ejemplo: al leer el elemento array[5], el caché carga también array[6], array[7]... porque es probable que el código también los necesite.

El kernel de Linux usa la RAM como caché del disco (page cache).

El Page Cache de Linux: cómo el SO usa la RAM como caché de disco

En Linux moderno (kernel ≥ 2.4), el page cache es el mecanismo central de caching de E/S. Cuando lees un archivo por primera vez, el kernel lo carga del disco y lo guarda en RAM. La segunda vez que lees ese archivo — incluso desde otro proceso — el kernel sirve los datos directamente desde RAM sin tocar el disco.

Algoritmos de reemplazo: ¿qué bloque se expulsa cuando el caché está lleno?

LRU — Least Recently Used

Se expulsa el bloque que lleva más tiempo sin accederse. Aprovecha la localidad temporal. Es el algoritmo más común en sistemas de archivos Linux (ext4, XFS, Btrfs). Requiere mantener un registro de tiempo de último acceso.

LFU — Least Frequently Used

Se expulsa el bloque menos frecuentemente accedido. Útil cuando ciertos datos se acceden repetidamente durante períodos largos. Problema: datos "envejecidos" con muchos accesos históricos pueden ocupar espacio innecesariamente.

FIFO — First In, First Out

El primer bloque en entrar es el primero en salir. Simple de implementar pero ignora completamente el patrón de acceso. Puede expulsar bloques muy activos si llevan mucho tiempo en caché.

ARC — Adaptive Replacement Cache

Algoritmo adaptativo que combina LRU y LFU dinámicamente. Usado en ZFS. Se adapta automáticamente a la carga de trabajo actual, comportándose como LRU para accesos únicos y LFU para accesos repetidos.

El cuello de botella migró al software: el problema NVMe

Una investigación reciente de USENIX FAST 2025 reveló algo contraintuitivo: con SSDs NVMe ultra-rápidos, el cuello de botella ya no es el hardware sino el propio page cache del kernel. La asignación de páginas de memoria para el caché consume entre un 24% y 36% de todos los ciclos de CPU durante operaciones de E/S pesada. El software del kernel, diseñado décadas atrás para discos lentos, se convirtió en el nuevo limitante.

💡
Consecuencia práctica

Bases de datos como PostgreSQL y MySQL permiten usar O_DIRECT para saltarse el page cache completamente y hacer E/S directa. Esto evita la doble copia (disco→page cache→buffer propio de la BD) y la sobrecarga del kernel, a costa de perder la transparencia del caching del SO.
